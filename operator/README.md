## Prerequisites

Before applying these manifests:

1. **Altinity ClickHouse Operator 0.26.0+** installed cluster-wide.
2. **Vault Agent Injector** (HashiCorp Vault 1.3.0+ with injector webhook) running and configured with Kubernetes auth.
3. **StorageClasses** available:
   - `standard` ‚Äî general-purpose SSD (Keeper data, ingest data, cache metadata directory)
   - `nvme-storageclass` ‚Äî NVMe/SSD-backed PVs (cache write-through disk at `/var/lib/clickhouse/s3_cache/`)
   Replace these names in the YAMLs to match your environment.
4. **Node labels and taints** applied to dedicated node groups (minimum counts):
   - Keeper nodes (‚â•5): `clickhouse/role=keeper` + taint `clickhouse/keeper=true:NoSchedule`
   - Ingest nodes (‚â•2): `clickhouse/role=ingest` + taint `clickhouse/ingest=true:NoSchedule`
   - Cache nodes (‚â•14): `clickhouse/role=cache` + taint `clickhouse/cache=true:NoSchedule`
   One cache pod per node is enforced by pod anti-affinity (each pod needs its own NVMe PV).
5. **Vault** configured with Kubernetes auth method, policy and role for ClickHouse (see `vault/serviceaccount.yaml`).
6. **All blocking Open Decisions resolved** ‚Äî see section below.

## Deployment Order

```bash
# 1. Namespace
kubectl apply -f 00-namespace.yaml

# 2. ServiceAccount ‚Äî must exist before pods start (Vault Agent needs it)
kubectl apply -f vault/serviceaccount.yaml

# 3. Keeper ‚Äî wait for quorum healthy before proceeding
kubectl apply -f 01-keeper.yaml
kubectl -n clickhouse rollout status statefulset -l app.kubernetes.io/component=keeper
# Verify quorum: exec into any keeper pod and run:
#   clickhouse-keeper-client -h localhost -p 2181 -q "ruok"  # expect "imok"

# 4. Main ClickHouseInstallation (ingest + cache clusters)
kubectl apply -f 02-chi.yaml
# Monitor: kubectl -n clickhouse get clickhouseinstallations -w

# 5. Client-facing Services
kubectl apply -f 03-services.yaml

# 6. NetworkPolicy
kubectl apply -f 04-networkpolicy.yaml
```

After all pods are Running/Ready, run DDL from [`clickhouse/`](../clickhouse/) in the order documented in `clickhouse/README.md`.

## PoC Validation Checklist (Phase 2 ‚Äî before scaling to 7√ó2)

Deploy a 1-shard √ó 2-replica cache PoC first. Validate all items before scaling.

- [ ] Keeper quorum healthy: `ruok` returns `imok` on all 5 nodes
- [ ] `SELECT * FROM system.clusters` on any cache pod shows both `ingest` and `cache` clusters in `remote_servers` (auto-generated by Altinity operator ‚Äî architecture.md ¬ß7.1)
- [ ] `SELECT * FROM system.disks` shows disk type `s3` (NOT `s3_plain`) for `s3_main`
- [ ] Zero-copy replication: INSERT into Replica 0 ‚Üí part appears on Replica 1 with no pod-to-pod byte transfer (`SELECT * FROM system.replication_queue` drains; `transfer_bytes` stays 0)
- [ ] S3 parts appear under correct shard prefix: `s3://bucket/clickhouse/cache/shard-1/`
- [ ] NVMe cache populated on write: `SELECT * FROM system.filesystem_cache` shows non-zero `used_size`
- [ ] TTL drop: populate a part where all rows have `eviction_date < today`; within 60 s the part is dropped and S3 objects are physically removed
- [ ] Compliance DELETE load test: `DELETE FROM t WHERE user_id = X` + `OPTIMIZE TABLE PARTITION FINAL` on a production-sized partition; measure elapsed time and confirm against 24 h SLA (input for Open Decision #11)
- [ ] Patch parts (compliance UPDATE): `ALTER TABLE t UPDATE` with zero-copy enabled; both replicas reflect the update
